# 문서 카테고리 분류 기준 (lostark-remote-kakao)

## 목적

`changes/` 남용을 줄이고, 문서의 주된 목적에 따라 카테고리를 일관되게 선택하기
위한 기준입니다. 본 서비스는 TypeScript ESM Fastify 기반 HTTP wrapper (공식
로스트아크 API 의 캐시·재노출 계층) 이며 웹 UI/관리자 UI 가 없으므로,
`uiux-qa-report/` 같은 카테고리는 존재하지 않습니다.

## 핵심 원칙

- 카테고리는 "무엇이 바뀌었는가" 보다 **"이 문서를 나중에 왜 다시 찾는가"** 를
  기준으로 선택합니다.
- 한 문서에는 하나의 주 목적만 둡니다.
- 문서가 두 목적을 동시에 가지면, 오래 참조할 기준 문서를 우선 만들고 필요할
  때만 보조 문서를 추가합니다.

## 카테고리 선택 기준

| 카테고리        | 이럴 때 사용                                                                                                                                                                                                               | 이럴 때는 사용하지 않음                                                                     |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| `adr/`          | 기술 선택/아키텍처 원칙에 대한 **불변 결정 레코드**. Fastify·pg(PostgreSQL)·Yarn Berry 채택, 패키지 분할 경계, 캐시 3-tier 정책 같은 되돌리기 어려운 결정                                                                  | 단건 구현 변경, 일회성 분석                                                                 |
| `analysis/`     | 코드/데이터 흐름/성능/리스크를 이해하거나 원인을 추적하는 문서. PostgreSQL 쿼리 플랜 검토, 공식 로스트아크 API 레이트리밋 분석, normalizer 성능 비교 등                                                                    | 수정 결과 changelog, 장기 운영 기준, 계약 스펙                                              |
| `architecture/` | 시스템 경계, 레이어 규칙(shared / data-service / rest-service / udp-service), 저장 구조(PostgreSQL 스키마 + Redis 키 prefix), 데이터 흐름(3-tier 캐시), Fastify 라우트 토폴로지처럼 오래 유지될 구조 설명                  | 단건 변경 이력, 일회성 조사, ADR 에 가까운 결정(→ `adr/`)                                   |
| `backlog/`      | 미채택 자동화/기능 아이디어, 재검토 트리거 조건 기록. UDP 게이트웨이 확장안, 추가 캐시 레벨 도입 검토 등                                                                                                                   | 확정된 로드맵, 완료된 작업 기록                                                             |
| `changes/`      | 이미 반영된 구현 변경의 이력(changelog). REST 라우트 동작, normalizer 로직, 캐시 키/TTL, PostgreSQL 스키마, 공식 API 클라이언트 계약이 실제로 바뀐 경우                                                                    | 분석만 한 경우, 문서만 동기화한 경우, 재사용 가이드/절차, 장애 조사만 있고 수정이 없는 경우 |
| `contracts/`    | 외부/내부 **계약 기준 문서**. 공식 로스트아크 API 호출 계약(developer-lostark.game.onstove.com 의 V7/V8/V9 응답 envelope), 자체 REST/OpenAPI 계약(LoA-Bot 등 다운스트림이 소비)                                            | 특정 날짜의 변경 이력, 조사 메모                                                            |
| `development/`  | 반복해서 재사용할 개발 규칙, 절차, 워크플로우, 툴 운영 기준. 이 문서 자체도 여기 속함. Yarn Berry PnP 운용, 모노레포 의존 검증 절차 등                                                                                     | 특정 구현 변경 이력, 단건 장애 보고서                                                       |
| `domain/`       | **로스트아크 게임 도메인 지식**. 레이드 난이도 체계, 전투력 계산식, 각인/카드/보석 시스템, 시블링/캐릭터 관계, 카오스 게이트/필드보스 스케줄 규칙                                                                          | 구현 변경 이력, 서비스 내부 구조                                                            |
| `issues/`       | 운영 이슈, 장애, 재현 조건, 영향 범위, 원인, 대응 내역. REST 서버 크래시, 라우트 오동작, Redis/PostgreSQL 장애, 공식 API 장애·rate-limit 폭주 대응                                                                         | 일반 기능 변경 기록, 장기 가이드                                                            |
| `maintenance/`  | 운영자가 따라야 하는 점검, 수동 조치, 배치, 데이터 정리, 유지보수 절차. PostgreSQL 마이그레이션 실행 (`yarn db:migrate`), `yarn workspace @lostark/rest-api dump:openapi` 재실행, 캐시 초기화/복구, loa-platform compose 재기동 | 개발 규칙, 분석 보고서                                                                      |
| `security/`     | 인증/인가(API 키), 입력 검증(Fastify route schema + zod), 비밀 값 관리(`.env`), `@fastify/helmet`/`@fastify/cors` 정책, 보안 정책/패턴을 설명하는 기준 문서                                                                | 특정 기능 수정 이력 자체                                                                    |
| `work-log/`     | 작업 중간 기록, 세션 간 handoff 메모, 시점성 있는 진행 로그 (예: 본 인프라 이식 작업 로그)                                                                                                                                 | 최종 기준 문서, 장기 참조용 설명                                                            |
| `graph/`        | **자동 생성** (graphify). 사람이 손으로 편집하지 않음. `index.md` 만 메타로 커밋                                                                                                                                           | 사람이 작성하는 아키텍처 설명(→ `architecture/`)                                            |

## `changes/` 를 써도 되는 경우

- 코드가 실제로 수정되었다.
- 또는 코드가 소비하는 설정/정적 데이터(env 해석 규칙, 캐시 TTL 상수, route
  schema) 가 바뀌어 실제 동작이 달라졌다.
- 문서의 중심이 "현재 구조 설명" 이 아니라 "이번 작업에서 무엇이 왜 바뀌었는지"
  다.
- 읽는 사람이 "언제 어떤 구현 변화가 들어갔는지" 를 확인하려고 이 문서를 찾는다.

## `changes/` 로 보내면 안 되는 경우

- 구현은 그대로고 설명만 보강했다.
- README / CLAUDE.md 와 docs/ 를 동기화했다.
- 재사용 가능한 작업 절차나 작성 규칙을 정리했다(→ `development/`).
- 버그 원인/구조를 분석했지만 수정은 아직 없다(→ `analysis/`).
- 운영 장애를 복기하고 재현/영향 범위를 남기는 것이 핵심이다(→ `issues/`).

## 빠른 결정 순서

1. 기술 선택/아키텍처 원칙에 대한 되돌리기 어려운 결정인가? → `adr/`.
2. 운영 장애나 이슈 대응 기록인가? → `issues/`.
3. 로스트아크 게임 자체에 대한 지식인가? → `domain/`.
4. 외부(공식 로스트아크 API) / 내부(자체 REST/OpenAPI) 계약 스펙인가? →
   `contracts/`.
5. 현재 기준의 구조/가이드를 설명하는 문서인가? → `architecture/`,
   `development/`, `security/` 중 하나.
6. 조사, 비교, 원인 추적, 리스크 파악이 핵심인가? → `analysis/`.
7. 수동 운영 절차나 유지보수 실행 가이드인가? → `maintenance/`.
8. 진행 중인 작업 메모인가? → `work-log/`.
9. 미채택 아이디어/흔적인가? → `backlog/`.
10. 위가 아니고 실제 구현 변경 이력을 남기는 문서인가? 그때만 → `changes/`.

## 함께 작성하는 규칙

- 구현 변경과 장기 기준 문서가 동시에 필요하면, 기준 문서는 해당 도메인
  카테고리(`contracts/`, `architecture/`, `domain/` 등) 에 두고 `changes/` 에는
  변경 요약만 남깁니다.
- 문서 동기화만 했다면 원칙적으로 `changes/` 를 만들지 말고, 대상 기준 문서를
  직접 갱신합니다.
- 장애 수정이 있었더라도 핵심 가치가 장애 원인과 대응 이력 보존이라면 `issues/`
  를 우선하고, 필요할 때만 관련 `changes/` 를 추가합니다.
- ADR 에 해당하는 결정이면 `changes/` 대신 `adr/` 에 불변 레코드로 남깁니다.

## 자주 헷갈리는 사례

- 공식 로스트아크 API 의 V9 ArmoryProfile 응답 envelope 가 바뀌고 우리
  normalizer/타입도 갱신됨: 기준 문서는 `contracts/` (공식 API 호출 계약), 변경
  이력은 필요할 때만 `changes/`.
- 자체 REST 라우트의 응답 스키마가 바뀜 → OpenAPI dump 갱신: 기준 문서는
  `contracts/` (자체 OpenAPI), 변경 이력은 `changes/` 와 `maintenance/`
  (다운스트림 재생성 절차) 양쪽 모두 가능 — 주 목적에 맞춰 선택.
- pg raw SQL 작성 컨벤션 + 마이그레이션 파일 작성 컨벤션을 정리함:
  `development/`.
- 특정 라우트의 캐시 동선/TTL 결정을 읽고 리스크를 정리함: `analysis/`.
- REST 서버 프로세스가 OOM 으로 재시작된 원인과 재현 조건을 남김: `issues/`.
- 로스트아크 군단장 레이드의 관문/난이도/입장 제한을 정리함: `domain/`.
- PostgreSQL 마이그레이션을 운영 환경에서 수동 실행하는 절차를 문서화함:
  `maintenance/`.
- Yarn Berry (PnP, strict) 를 고른 이유와 npm/pnpm 대안 비교를 영구 보존:
  `adr/`.
- UDP 게이트웨이의 신규 도메인 채널 도입을 검토했으나 현재는 보류: `backlog/`.
- LoA-Bot 측 `openapi-typescript` 재생성 절차의 표준화: `maintenance/` (운영
  절차) 또는 `contracts/` (계약 갱신 규약) — 문서 중심에 따라.
